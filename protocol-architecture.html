System Architecture

Module Reference

Each module runs as an independent HTTP server on its own port. They communicate with the ORCHESTRATOR, which coordinates the full pipeline. Modules have no hard dependencies on each other — you can run them individually or in any combination.

Core

🧠 MEMSTORE :8500

Persistent agent memory with two tiers. Working memory stores active context in-memory for fast recall. Long-term memory persists to SQLite across sessions. Automatic compression prevents context window overflow.

POST /save · GET /recall · DELETE /clear
Core

📋 TRAKR :8501

Cryptographic audit trail with SHA-256 Merkle chaining. Every action creates a tamper-evident entry. Verify the integrity of the entire chain at any time. SIEM-compatible output for enterprise compliance.

POST /append · GET /verify · GET /chain
Core

🛡️ GUARDRAIL :8502

Policy and governance engine. Define which actions are allowed or blocked per agent. Built-in default-deny policy for read-only safety. Logs every decision with reasoning for audit compliance.

POST /check · POST /policy · GET /policies
Core

📊 VITALS :8503

Real-time observability and telemetry. Tracks errors, latency, and throughput per agent. Streams events for external monitoring integration (Prometheus, Datadog, custom). Anomaly detection on metric drift.

POST /event · GET /metrics · GET /health
Core

💊 HEALER :8504

Autonomous incident recovery. Monitors module health and agent behavior. On failure detection, executes a configured strategy: restart process, scale up, failover to replica, graceful degradation, or human escalation.

GET /status · POST /heal · GET /strategies
Core

⚙️ ORCHESTRATOR :8505

Runtime orchestrator that ties the pipeline together. Routes incoming agent actions through GUARDRAIL → MEMSTORE → TRAKR → VITALS → HEALER in sequence. Exposes a unified health endpoint and coordinates cross-module workflows.

POST /execute · GET /modules · GET /health

Data Flow

When an agent sends an action through PROTOCOL, it passes through each module in sequence. The ORCHESTRATOR manages the full lifecycle.

1

ORCHESTRATOR receives action

Agent sends a POST /execute request with agent_id, action, payload, and framework identifier.

2

GUARDRAIL validates

Action checked against agent-specific policies. Blocked actions return immediately with reason. Allowed actions proceed with a pass token.

3

MEMSTORE enriches context

Relevant memories from previous sessions are injected into the action context. Agent receives persistent knowledge without manual recall.

4

TRAKR records

Action, timestamp, agent identity, and outcome appended to the Merkle chain. Evidence: the hash changes. Integrity: always verifiable.

5

VITALS streams telemetry

Action emitted as a telemetry event. Latency, error rate, throughput updated in real time. Available for external monitoring ingestion.

6

HEALER evaluates

Outcome checked against health thresholds. If the action failed or triggered an anomaly, HEALER decides whether to: do nothing, restart a module, or escalate.
